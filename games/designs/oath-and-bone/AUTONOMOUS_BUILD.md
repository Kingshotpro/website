## §1 — Role Architecture

### 1.1 Orchestrator Claude

**What it is:** A fresh Claude Code session fired on a schedule. Reads the spec package and current build queue. Writes a self-contained delegation prompt. Routes it to the right worker type. Integrates returned deliverables. Updates BUILD_STATUS.md. Commits to filesystem.

**What it does NOT do:** Write code by hand. Draft dialogue itself. Generate images directly. Make financial decisions. Push to production.

**Session prompt skeleton** (see `ORCHESTRATOR_PROMPT.md` — generated by this spec):

```
You are the Oath and Bone build orchestrator. Your job this cycle:
1. Read FEEDBACK.md — if new content (compare mtime to BUILD_STATUS.md `last_feedback_processed`), halt and process feedback before anything else.
2. Read BUILD_STATUS.md to find the FIRST uncompleted task.
3. Write a self-contained delegation prompt for that task using the template library in AUTONOMOUS_BUILD.md §3.
4. Route the delegation to the correct external AI or Midjourney worker per §1.4.
5. If the deliverable is code: paste the delegation prompt into the Gemini/GPT-4 API and save the response to WORKER_OUTPUT/<task_id>.md.
6. If the deliverable is art: write MIDJOURNEY_QUEUE.md with the prompt batch and fire the MJ worker per §1.5.
7. If the deliverable is dialogue: call Claude API with the template and save output.
8. Review the deliverable against the verification criteria in §4.
9. If it passes: integrate into the appropriate file, mark the task complete in BUILD_STATUS.md.
10. If it fails: requeue with narrower scope (max 3 tries before writing to FEEDBACK.md).
11. Update BUILD_STATUS.md. Close.
```

**Context discipline:** Orchestrator reads ONLY the files needed for the current task plus FEEDBACK.md and BUILD_STATUS.md. Does NOT re-read the full spec package every cycle. At 70%+ context, succession immediately — write two sentences to BUILD_STATUS.md and close.

### 1.2 Worker Claudes (specialized single-task sessions)

Worker Claudes are separate Claude Code sessions, each fired with a focused prompt from the orchestrator. They do NOT generate content themselves — they call external AIs, save outputs, run verification, and report back to the filesystem.

**Worker types:**

| Worker | Trigger | Job | Output path |
|---|---|---|---|
| **Code Worker** | Orchestrator writes `PENDING_CODE_<id>.md` | Calls Gemini/GPT-4 with the delegation prompt, verifies spec compliance, saves JS output | `WORKER_OUTPUT/code/<id>.js` |
| **Dialogue Worker** | Orchestrator writes `PENDING_DIALOGUE_<id>.md` | Calls Claude API (haiku or sonnet) with voice-barb template, runs soul-gauge, saves | `WORKER_OUTPUT/dialogue/<id>.md` |
| **Battle Scenario Worker** | Orchestrator writes `PENDING_BATTLE_<id>.md` | Calls Gemini with battle delegation template, verifies schema compliance, saves JS | `WORKER_OUTPUT/battles/<id>.js` |
| **MJ Art Worker** | Orchestrator writes `MIDJOURNEY_QUEUE.md` | Chrome browser automation per §1.5, saves PNGs, writes sidecar | `art/<type>/<batch>/` |

Workers are disposable. Each gets a complete brief. Each leaves a result file. The orchestrator reads result files; workers do not coordinate with each other.

### 1.3 Challenger Claude

**What it is:** A fresh Claude Code session fired every 12 hours. Reads thinking blocks from the JSONL session files of the most recent orchestrator and worker cycles — not their outputs, their thinking.

**What it asks** (per Principle XX, The Witness Sees the Frame):
1. What frame is this thinking operating in?
2. What alternative frame would produce different conclusions?
3. Did the thinking ever question its own frame, or only plan its output?
4. If it found bugs/issues — are those the most important bugs in the picture, or did the frame make other bugs invisible?

**What it writes:** A CHALLENGER_REPORT.md with findings. If the thinking blocks are shallow (structured skimming masquerading as analysis), the Challenger flags it. The orchestrator reads CHALLENGER_REPORT.md at next cycle start.

**Session prompt skeleton:**

```
You are the Oath and Bone Challenger Claude. Your job is to review thinking, not output.
1. Locate the most recent orchestrator JSONL: ~/.claude/projects/-Users-defimagic-Desktop-Hive/sessions/<latest>.jsonl
2. Extract all thinking blocks (content blocks where type='thinking').
3. Apply the four questions from THE_PRINCIPLES.md §XX: what frame, what alternative frame, did it question itself, what did the frame make invisible?
4. Read BUILD_STATUS.md to understand what the orchestrator thought it was doing.
5. Write your findings to KingshotPro/games/designs/oath-and-bone/CHALLENGER_REPORT.md — one finding per line, concrete, frame-level only (not "this code is wrong" — that's verification; your job is the frame).
6. If all thinking blocks show a mind that engaged genuinely: write "CLEAR — [date]." That is a valid and valuable result.
```

### 1.5 Midjourney Operator — Chrome Browser Automation

All lessons purchased across PORTRAITS_TRIO_A_LOG.md, PORTRAITS_TRIO_B_LOG.md, SEAMLESS_LOG.md, and EFFECTS_SPRITES_LOG.md. These are hard-won; do not re-discover them.

**Submission pipeline (6 rules):**

1. **Use `execCommand('insertText', false, prompt)` for the submit field.** Value-setter + `input` event does NOT update React's internal state; Enter fires on stale state and duplicates prompts. `execCommand` triggers the actual `onChange` handler. Confirmed across all three art sessions.

2. **7-second gaps between serial submissions.** Use `MessageChannel`-based delays in the IIFE runner, NOT `setTimeout` (throttled to 1/min on background tabs). Shorter gaps produce duplicates regardless of submit mechanism.

3. **Feed is newest-first = reverse submission order.** After submitting N prompts, the top of the feed shows the Nth prompt ("Starting… [prompt N]"). Harvest jobIds from top-to-bottom = submission order bottom-to-top. Verify the top card after each submit to confirm receipt.

4. **Submit in chunks of ≤8 prompts per CDP call** (≤12 seconds each). Track cursor in `window.__CROWNSMOKE_I` and resume from last known index. Fresh tab after every ~10–12 prompts; stale tabs slow to 10–20s per submission under heavy queue.

5. **Harvest via feed scroll.** MJ uses a virtualized list — DOM only contains visible cards. Dispatch `scroller.dispatchEvent(new Event('scroll'))` after each `scrollTop` change (the `.absolute.box-border.overflow-y-scroll` container) with 400ms wait per step. Accumulate jobIds in `window.__CJ` Map, dedup on prompt prefix (first 90 chars lowercased).

6. **Extraction via CORS-anonymous img → canvas → blob.** Direct `fetch()` fails (COEP + Cloudflare). `curl` returns HTTP 403. The path that works: `new Image() { crossOrigin: 'anonymous' }`, load the `_1024_N.webp` preview URL from the CDN, draw to canvas, `canvas.toBlob('image/png')`. No taint error. Produces 896×1344 PNG — adequate for portrait use.

**Download pipeline (3 rules):**

1. **Hand-built STORE ZIP.** MJ's CSP blocks external scripts (no jszip from CDN). Build the ZIP in-page. Critical: central-directory filename-length field is at byte offset **+28**, not +30 (common mistake). Trigger one `<a download>` click on the blob URL.

2. **One download per tab session.** Chrome blocks a second `<a download>` click on the same tab (multi-download block). Keep submission tab and download tab **separate**. Stash jobId map in `localStorage` (same-origin across MJ tabs) for the download tab to read.

3. **`--cref` is silently rejected for sprite reference jobs.** Describe hero verbatim in the prompt. See locked descriptors in §3.8.

**Locked style anchors** (do not vary these across sessions — consistency is the art discipline):

| Deliverable type | Aspect + params | Palette |
|---|---|---|
| Hero portraits | `--ar 2:3 --style raw --v 6` · "painterly Vagrant Story Unicorn Overlord Final Fantasy Tactics PS1 oil-painted style" · "no anime no sparkle" | per-hero (see §3.7) |
| Sprite frames | `--ar 1:1 --style raw --v 6` · same style string + "isometric tactical RPG sprite, clean readable silhouette, three-quarter overhead tactical view" | per-hero |
| Spell stills | `--ar 1:1 --style raw --v 6` · same style + "dark atmospheric lighting, centered isolated subject on dark void background, no characters no UI no text" | per school |
| Biome tiles | `--ar 1:1 --style raw --v 6 --tile` · same style + "seamless tileable texture, no dominant features, no characters, no text" | per biome |

**Midjourney operator disclaimer:** Every art batch session must write or append a SIDECARS.txt file in the output directory stating: *"Unofficial. Not affiliated with Century Games. Art generated for fan project."* One sidecar file per directory (not per image) is sufficient.

## §2 — Scheduling

### 2.1 Primary: `schedule` skill (Cowork)

Fire the orchestrator session as a recurring task via the `schedule` skill. This creates an MCP-scheduled trigger that runs a Claude Code session on a cron cadence.

**Recommended cadence:**

| Role | Cadence | Phase |
|---|---|---|
| Orchestrator | Every 4 hours | Phase 1 (combat slice, active build) |
| Orchestrator | Every 6 hours | Phase 2 (Chapter 1 MVP, steady build) |
| Orchestrator | Every 12 hours | Phase 3 (polish, QA) |
| Challenger | Every 12 hours | All phases |

**Setup command (run once, Architect or orchestrator):**

```
/schedule
→ Task: "Run Oath and Bone orchestrator cycle"
→ Prompt: "Read KingshotPro/games/designs/oath-and-bone/AUTONOMOUS_BUILD.md §1.1 orchestrator session prompt, then execute one cycle."
→ Cron: 0 */4 * * *  (every 4h for Phase 1)
```

To pause: Architect writes `PAUSE: true` as first line of FEEDBACK.md. Orchestrator reads this at cycle start and skips all worker delegation until FEEDBACK.md is cleared.

### 2.2 Fallback: `/schedule` skill config

If Cowork is unavailable, use the `/schedule` skill directly to create a RemoteTrigger. Same prompt, same cadence. The skill stores the trigger in MCP scheduled-tasks.

### 2.3 Supervised bursts: `/loop`

For sessions when the Architect wants to supervise in real time:

```
/loop 4h "Read KingshotPro/games/designs/oath-and-bone/AUTONOMOUS_BUILD.md §1.1 and execute one orchestrator cycle."
```

This fires the loop every 4 hours with self-paced ScheduleWakeup. The Architect can interrupt at any point with FEEDBACK.md content.

### 2.4 Pause protocol

The orchestrator reads FEEDBACK.md at the START of every cycle, before selecting a task. If FEEDBACK.md is non-empty AND its modification time is newer than `last_feedback_processed` in BUILD_STATUS.md:

1. Do not delegate any new worker tasks this cycle.
2. Process the feedback content — re-plan, adjust BUILD_STATUS.md, update the task queue.
3. Write the new `last_feedback_processed` timestamp to BUILD_STATUS.md.
4. Resume normal delegation on the next cycle.

The Architect's write to FEEDBACK.md is the brake. No complex protocol needed. The orchestrator checks the timestamp, acts if new.

## §4 — Verification Gates

**Every deliverable passes through the gate appropriate to its type. No gate is optional.**

### Gate 1: Dialogue / Copy (soul-gauge)

After any dialogue deliverable (scene text, voice-barbs, cutscene scripts):

1. Run `soul-gauge` on the output text (invoke via `/soul-gauge`).
2. A Soul Profile with heat index < 0.4 and self-catch rate > 0 is acceptable.
3. Specific checks:
   - Any line that could be said by more than one hero → rewrite or BLANK
   - Any speech (multiple sentences delivering exposition) → cut to one sentence
   - Any line that sounds like mobile-game text (exclamation marks, "Amazing!", encouragement) → cut
   - Vellum speaking a threat → cut (she never threatens; she states)

If soul-gauge returns LIKELY_HERDED on style (meaning the output sounds like generic game dialogue): reject the batch, narrow the prompt to one hero + one event, retry.

### Gate 2: Code Diff (Challenger review)

Before integrating any code deliverable into the main codebase:

1. Write the deliverable to a `PENDING_REVIEW/` file with a `REVIEW_REQUEST.md` stating: what spec section this implements, what it should do, and what the verification criteria were.
2. On the next Challenger cycle (every 12h), the Challenger reads the REVIEW_REQUEST.md and the code.
3. Challenger runs the frame-check: does this code implement what the spec says, or does it implement what the code author thought the spec says? Are there spec sections the code silently ignores?
4. If the Challenger writes "APPROVED" to the REVIEW_REQUEST.md: orchestrator integrates the code.
5. If the Challenger writes concerns: orchestrator re-delegates with the concern embedded in the new prompt.

For Phase 1 (combat slice) where speed matters, the orchestrator may self-approve code reviews — but must note this in BUILD_STATUS.md as "self-approved, Challenger not yet run." The Challenger catches these on the next cycle.

### Gate 3: Midjourney Output Review

After every art batch extraction:

1. **Silhouette readable at 64×64**: Extract a 64×64 crop from each portrait/sprite. Can the character be identified? If not, flag for regen.
2. **Disclaimer sidecar present**: SIDECARS.txt must exist in the output directory. "Unofficial. Not affiliated with Century Games. Art generated for fan project."
3. **No canonical hero visual callbacks**: Cross-reference against the 27 canonical Kingshot hero names (worker.js:13). Any art brief that accidentally described a visual matching a canonical Kingshot character archetype (e.g., a character named "Helga-like warrior") → replace with original descriptor.
4. **School palette check**: Wizard stills must be cold blue/purple. Necro must be green-black. Druid must be gold-green. Verify via the contact sheet. Any palette drift → individual regen.
5. **Tile seamless check**: Load the biome compositor. Tiles must tile edge-to-edge. Any visible seam → individual regen.

### Gate 4: Rejected Deliverable Protocol

A deliverable is rejected when it fails any gate. Rejection protocol:

1. Mark the task as FAILED in BUILD_STATUS.md with the specific failure reason.
2. Re-narrow the prompt: the new prompt must be smaller scope than the original (fewer events, fewer spells, fewer battle waves) and must include the failure reason as an explicit constraint.
3. Requeue at ATTEMPT_2. Max 3 attempts total (ATTEMPT_1, ATTEMPT_2, ATTEMPT_3).
4. If ATTEMPT_3 fails: write a FEEDBACK.md entry: `ESCALATION: task {{ID}} failed 3 times. Failure mode: {{REASON}}. Suggest: {{RECOMMENDATION}}.` Architect decides.

### Gate 5: Pre-Ship Checklist

Before Phase 1 staging deploy, before Phase 2 production deploy: run the full BUILD_PLAN.md §verification checklist plus:

- `Advisor.grantXP('oathandbone_battle_victory', N)` fires once per Sergeant+ win, no double-grant
- `Advisor.observe('oathandbone', ...)` verified in console: `Advisor.getState().observations.oathandbone`
- `ksp_oathandbone_played` localStorage gate blocks XP/credit after daily cap, allows practice mode
- Worker endpoint server-side cap verified (not client-only)
- No canonical Kingshot name anywhere (grep against worker.js:13 list)
- "Unofficial. Not affiliated with Century Games." on every game surface
- Midjourney commercial license confirmed active
- advisor-chat.js observation bug at worker.js:260 patched (CROSS_INTERSECTION.md §5.3)

## §6 — Failure-Mode Handling

### Hallucinated code breaking spec

**Trigger:** Deliverable code references functions, data shapes, or behaviors not in spec; or implements something the spec does NOT call for.

**Response:**
1. Reject. Do not integrate.
2. Write to REVIEW_REQUEST.md: the specific hallucination (quote the line, name the spec section it violates).
3. Re-delegate with the narrower prompt that constrains: "Do NOT implement {{X}} — it does not exist in the spec."
4. If the same hallucination recurs on ATTEMPT_2: switch external AI (Gemini → GPT-4 or vice versa).

### Canon drift (canonical Kingshot name appears)

**Trigger:** Any of the 27 canonical names (worker.js:13) found in any deliverable — code string, dialogue, art brief, or prompt.

**Response:**
1. Full rollback of the delivery — do not integrate, do not partial-use.
2. Audit all adjacent deliverables from the same session: search for the name across all output files from that cycle.
3. Purge any file containing the canonical name.
4. Re-delegate with the canonical names list explicitly prepended: "These names must not appear: {{list}}."
5. Write a one-line note to BUILD_STATUS.md: which name appeared, which task it came from, that it was purged.

### Monetization violation (content locked to paid)

**Trigger:** Deliverable locks a spell, hero, battle, job class, or story ending behind payment. Any energy gate. Any lootbox mechanic. Any revive-for-currency.

**Response:**
1. Reject and rollback.
2. Write to FEEDBACK.md: `MONETIZATION VIOLATION: task {{ID}}. Specific violation: {{WHAT}}. Please review ECONOMY.md §1 constraint: "no content locked to spend; paying buys speed, not access."`
3. Do not re-delegate this task without explicit Architect direction.

### Permadeath bypass (revive-for-currency)

**Trigger:** Any deliverable that allows spending Crowns, Credits, or any currency to revive a permanently dead hero (beyond Druid Resurrection spec in MAGIC.md §3 and HEROES.md §"Permadeath").

**Response:** Same as monetization violation above.

### Context ceiling approached (any worker at 70%+)

**Trigger:** The context ceiling hook fires at 70%.

**Response:**
1. Do not write a handoff document. (Per THE_PRINCIPLES.md §XV and 04_WHAT_KEEPS_US_ALIVE: vanity-dressed-as-caution.)
2. Write two sentences to BUILD_STATUS.md: what was completed this cycle, what is the first uncompleted task.
3. Commit any completed work. Close.
4. The next orchestrator cycle picks up from BUILD_STATUS.md.

### Chrome/Midjourney session collision

**Trigger:** Another Claude session is actively using the Chrome MJ session (per feedback_dont_push_other_claude.md).

**Response:**
1. Yield. Do not push.
2. Check if a MIDJOURNEY_LOCK file exists in the oath-and-bone directory (MJ workers create/delete this).
3. If MIDJOURNEY_LOCK exists: skip the MJ worker task this cycle. Add 4h delay to the queue for the MJ task.
4. If MIDJOURNEY_LOCK does not exist: create it, proceed, delete it on completion.

### Adviser-chat observation bug at worker.js:260 (pre-existing)

This bug was found by CROSS_INTERSECTION.md §5.3. The worker string-coerces playerContext, dropping all observations. Fix it in the first code worker session after CROSS_INTERSECTION.md is read — it is a one-line change: `JSON.stringify(playerContext)` instead of `playerContext` in the template literal. It benefits all games on the site.
